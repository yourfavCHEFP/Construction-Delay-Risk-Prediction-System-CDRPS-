Things to try next time 🌱

None of these are "you did it wrong." They're "here's how to make it even better." Each card has what, why it matters, and what I'd do.

I1What does the "Delay Risk Index" actually mean?▶

What you did
The headline number is just the average of every numeric column in the row:

df["Delay_Risk_Index"] = df[feature_cols].mean(axis=1)

Why it matters
Here's the thing — those columns aren't on the same scale or even about the same thing. You might be averaging "years of experience" (a number like 12) with "severity rating" (1–5) and maybe even an ID column. The biggest numbers just bully the average, so the "risk" score can go up for reasons that have nothing to do with risk. If your mentor asked "why is this person high-risk?", right now it'd be hard to answer.

Suggestion
Decide what "risk" actually means, then build the number on purpose. Easiest win: average the scaled values (you already create a scaled version for the charts — reuse it!) so every column counts equally. Even better: pick just the columns that really relate to delay (the severity/likelihood questions) and write a one-line comment saying what the index is.
I2Your cluster colours are blending together▶

What you did
On the PCA scatter plot you colour the dots by cluster:

fig = px.scatter(df, x="PC1", y="PC2", color="Cluster", ...)

Why it matters
Because Cluster is a number (0, 1, 2, 3), Plotly thinks it's a measurement and colours it like a temperature scale — light blue to dark blue. So instead of four clearly different groups, you get four shades of the same colour and they're hard to tell apart. The whole point of a cluster plot is "look, these are different groups!" — and right now that's getting lost.

Suggestion
Tell Plotly to treat the cluster as a label, not a number, by turning it into text first: 
color=df["Cluster"].astype(str)
One tiny change and each group pops in its own distinct colour.
I3One of your cleanup tricks is on its way out▶

What you did
You convert columns to numbers with df.apply(pd.to_numeric, errors="ignore").

Why it matters
Two things. First, that errors="ignore" setting is being removed from newer pandas, so this'll start throwing warnings (and eventually break). Second — and more important — "ignore" means if a column can't be turned into numbers, pandas quietly hands it back unchanged and says nothing. So genuinely messy data can sneak through without you ever knowing.

Suggestion
Use errors="coerce" instead. Anything that isn't a number becomes a blank (NaN), and then you decide what to do — tell the user, drop it, or fill it in. The difference is "problems are visible" vs "problems are hidden," and visible always wins.
I4Why exactly 4 clusters?▶

What you did
KMeans is always set to find 4 groups, no matter what file gets uploaded: n_clusters=4.

Why it matters
4 might be perfect for one dataset and totally wrong for another — there's no single right answer baked into the data. Hardcoding it hides a decision that should really be visible. (Same idea with squishing everything down to just 2 PCA dimensions before clustering — you might be throwing away useful detail.)

Suggestion
Add a little st.slider in the sidebar so people can try different numbers of clusters, and/or look up the "elbow method" or "silhouette score" — they're simple ways to let the data suggest a good number for you.
I5You already have these tools — reuse them!▶

What you did
All the data-cleaning logic lives inside app.py. But the project already has ready-made helpers in CDRPS/src/preparation/ and CDRPS/src/ingestion/. There's also a leftover copy of the app at CDRPS/notebooks/app.py now sitting next to your new one.

Why it matters
When the same logic lives in two places, they drift apart — you fix a bug in one and forget the other. And mixing the "screen" code with the "data" code makes both harder to test. Two copies of app.py will also confuse the next person about which one is real.

Suggestion
Import the existing helpers from src and call them from the dashboard, so the app file mostly just handles the buttons and charts. Then delete (or clearly retire) the old notebook copy so there's one obvious source of truth.
I6No tests yet for the data crunching▶

What you did
The clever logic in process_data doesn't have any tests around it.

Why it matters
All those edge cases you handled (empty file, all-same columns, not enough columns) are exactly the kind of thing that quietly breaks later when someone tweaks the code. Tests are basically a safety net that yells at you the moment something regresses.

Suggestion
Pull process_data out into its own file and write a handful of quick pytest checks — one happy path with a tiny made-up table, plus one for each edge case. The repo already has a tests/ folder to drop them in.
I7Your tooltips are showing everything▶

What you did
The scatter plot uses hover_data=df.columns, which puts every single column into the hover popup.

Why it matters
On a wide survey that's a giant, cluttered tooltip where the useful bits get buried. Hovering should answer "what's important about this dot?", not dump the whole spreadsheet.

Suggestion
Just pick the few fields that matter, e.g. hover_data=["Delay_Risk_Index", "Cluster"] plus a couple of key survey questions.
I8You're quietly shrinking big files▶

What you did
If a file has more than 5,000 rows, you randomly keep 5,000 of them — but you don't tell the user.

Why it matters
Smart move for speed, but imagine uploading 20,000 rows and the results only describe a random quarter of them without any heads-up. That's a confusing surprise.

Suggestion
Pop up a quick st.info("Showing a random sample of 5,000 rows") when it happens, so people know what they're looking at.
