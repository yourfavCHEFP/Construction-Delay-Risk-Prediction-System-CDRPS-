🏗️ AI Construction Delay Prediction System

Functional Specification Author: Olumide [CHEF_P] · Date: 07 June 2026

1. Project Overview

This system applies machine learning techniques to predict the probability of schedule delays in civil engineering projects. The model analyzes historical construction data, material delivery logs, contractor performance records, and environmental factors such as rainfall intensity and temperature variations. The system outputs a delay‑risk score and provides interpretable insights to support proactive project planning and resource allocation.

Definition of Delay: For this project, a construction activity or project is considered "delayed" if the actual completion date exceeds the planned completion date by more than 10% of the planned duration (e.g., a 100‑day project delayed by >10 days). This threshold can be adjusted based on domain expert input.

The model is trained on historical labeled data where each project is tagged as "delayed" or "on‑time" based on the defined delay threshold.

2. Objectives

  • Predict the likelihood of schedule delay before and during project execution.
  • Identify the most influential factors affecting project timelines.
  • Provide a numerical risk score (0–100) and a categorical rating (Low, Moderate, High).
  • Offer an interactive dashboard for project teams.
  • Enable export of predictions and reports for documentation and compliance.

3. Users and Roles

  • Project Manager: Reviews delay predictions and monitors risk levels.
  • Site Engineer: Inputs updated field data such as progress and equipment status.
  • Data Analyst: Manages dataset quality and retrains the model.
  • Executive Stakeholder: Views high‑level summaries and portfolio risk distribution.

4. Functional Requirements

📥4.1 Data Input

  • System shall accept CSV or Excel uploads containing project attributes.
  • System shall validate required fields such as project_id, planned_duration, actual_progress, rainfall_mm, material_delivery_delay_days.
  • System shall notify users of missing or inconsistent data.

🤖4.2 Model Prediction

  • System shall generate a delay‑risk score between 0 and 100.
  • System shall classify risk as Low (0–30), Moderate (31–70), or High (71–100).
  • System shall highlight top contributing factors using feature importance or SHAP values.
  • System shall provide confidence levels based on model performance.

📊4.3 Reporting

  • System shall display predictions in a sortable, filterable table.
  • System shall generate visual charts (risk distribution, factor impact).
  • System shall allow export of results as CSV or PDF.
  • System shall allow users to download a summary report for each project.

Data Ingestion Module

The Data Ingestion Module is responsible for accepting, validating, and loading project datasets into the system. It ensures that only supported file formats are processed and that the dataset is structurally suitable for downstream validation and preparation tasks.

Supported File Formats

The system shall accept project data files in the following formats:

  • CSV (.csv)
  • Excel (.xlsx, .xls)

Any other file format shall be rejected with a clear, human-readable error message.

File Validation Requirements

Before loading any dataset, the system shall perform the following validation checks:

File Existence Check

If the file path does not point to an existing file, the system shall display:
“File not found at path: <path>. Please provide a valid file location.”

File Extension Check

If the file extension is not .csv, .xlsx, or .xls, the system shall display:
“Unsupported file format. Please upload a CSV or Excel file.”

File Extension Check

If the file exists but contains no data, the system shall display:
“The uploaded file is empty. Please upload a file containing project data.”

Zero-Row Check

If the file loads successfully but contains zero rows, the system shall display:
“The file was loaded but contains no rows. Please verify the dataset.”

Data Loading Requirements

  • The system shall load the file into a structured DataFrame-like object.
  • The system shall not expose raw error traces to the user.
  • The system shall return clean, structured error messages for any failure.

Data Preview Requirements

The system shall provide a preview function that:

  • Returns the first N rows of the dataset (default = 5).
  • Allows the user to specify a custom number of rows.
  • Returns an empty structure with a message if the dataset contains no rows.

Integration With Validation Module

After successful ingestion, the loaded dataset shall be passed to the Schema Validation Module. The ingestion module itself does not check for missing or inconsistent values; instead, it ensures the dataset is correctly loaded so that validation can proceed.

This supports the functional requirement:
“The system shall notify users of missing or inconsistent data.”

Data Validation Module

The Data Validation Module ensures that the uploaded dataset contains all required fields, adheres to expected structural rules, and does not contain missing or inconsistent values that would compromise downstream processing.

Required Column Validation

The system shall verify that all mandatory fields are present in the dataset. If any required column is missing, the system shall display:

“The dataset is missing required fields: <list_of_missing_columns>.”

The validation process shall not proceed until all required fields are present.

Missing Value Detection

The system shall analyze the dataset for missing values and produce a summary indicating:

  • Columns containing missing values
  • The number of missing entries per column

If missing values are detected, the system shall notify the user with a structured message summarizing the findings.

Inconsistency Detection

The system shall detect basic inconsistencies such as:

  • Negative values in fields that must be non-negative
  • Invalid ranges (e.g., planned duration ≤ 0)
  • Impossible values (e.g., rainfall < 0)

Any inconsistencies shall be reported in a structured format for user review.

Validation Output Structure

The validation module shall return a structured object containing:

  • List of missing required columns
  • Summary of missing values per column
  • List of detected inconsistencies

This output shall be used by the Data Preparation Module to determine the appropriate cleaning strategy.

5. Non‑Functional Requirements

  • Prediction response time should be under 2 seconds for up to 2,000 projects.
  • System should maintain 95% uptime for operational reliability.
  • The model should aim for an F1‑score in the range of 78–85% based on validation data, although final performance will depend on data quality, class balance, and feature availability.
  • System should ensure secure handling of project data using encryption.
  • Interface should be responsive and accessible across devices.

6. Assumptions and Limitations

  • Uploaded datasets are assumed to be accurate and complete.
  • The training dataset is expected to contain between 1,000 and 10,000 historical project records. The proportion of delayed vs. non‑delayed projects may be imbalanced; if so, techniques such as class weighting or oversampling will be applied to ensure model robustness.
  • The model predicts schedule delays only, not cost overruns or safety risks.
  • Weather predictions rely on historical averages and may not reflect extreme anomalies.
  • Model performance depends on the quality and quantity of available training data.
  • The system supports decision‑making but does not replace expert judgment.

System Flow Description

This section describes the end-to-end flow of data and operations within the Construction Delay Risk Prediction System. The flow outlines how user inputs and datasets move through the system’s modules to produce predictions and insights.

High-Level System Flow

  1. Dataset Upload: User uploads a CSV or Excel file containing project data.
  2. Data Ingestion: System validates file format, existence, and structure, then loads the dataset.
  3. Data Validation: System checks for required fields, missing values, and inconsistencies.
  4. Data Preparation: System cleans, encodes, scales, and splits the dataset.
  5. Model Training: System trains multiple machine learning models and selects the best performer.
  6. Model Evaluation: System computes evaluation metrics and stores results.
  7. Model Deployment: The selected model is saved and made available for predictions.
  8. Prediction: User submits project attributes; system preprocesses inputs and generates a delay risk prediction.
  9. Explainability: System generates SHAP-based explanations for the prediction.
